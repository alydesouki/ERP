# Orval generated query keys

Orval generates query keys prefixed by the **HTTP path**, not the operationId.
e.g. `getListUsersQueryKey()` returns `["/api/users", params?]`, not `["listUsers"]`.

**Rule:** to invalidate a list after a mutation, use the path prefix:
`queryClient.invalidateQueries({ queryKey: ["/api/users"] })`. invalidateQueries
does prefix matching, so this covers all param variants.

**Why:** Using a guessed key like `["listUsers"]` silently matches nothing — the
mutation succeeds (201) but the list never refetches. This passed typecheck and
only surfaced in e2e testing.

**How to apply:** After create/update/delete mutations (Orval hooks do NOT
auto-invalidate), call invalidateQueries with the resource's path prefix. Also
invalidate `["/api/audit-logs"]` after audited actions so the audit view refreshes.

# Passing `enabled` / options to generated query hooks

The generated `options.query` type is the full `UseQueryOptions` (NOT Partial), so
`queryKey` is **required** when you pass any query option. Provide it via the
matching getter, e.g.
`useListUsers(PARAMS, { query: { enabled, queryKey: getListUsersQueryKey(PARAMS) } })`.

# Setup-status gate

After a run-once setup completes, invalidate `getGetSetupStatusQueryKey()` before
navigating, or the cached `isSetupComplete:false` keeps the app stuck on the setup
screen until a hard reload.
