# Database Documentation

> All tables documented from direct inspection of `lib/db/src/schema/` and the production SQLite DB at `%APPDATA%/ShoeStorePOS/store.db`.

---

## Database Technology

- **Engine:** SQLite (via `libsql` ÔÇö Turso's WAL-optimized fork)
- **ORM:** Drizzle ORM with TypeScript type inference
- **ID Strategy:** `crypto.randomUUID()` ÔÇö UUIDs for all primary keys
- **Monetary Storage:** `text` (decimal string, e.g., `"1250.50"`) to avoid floating-point errors
- **Timestamps:** `integer` in millisecond epoch mode (`timestamp_ms`)
- **Soft Deletes:** Used on `users` (`is_deleted`), products (`is_active`)
- **Tenant Isolation:** Every table has `store_id` FK for multi-tenant readiness

## Schema Migration Strategy

The project uses **two migration mechanisms**:

### 1. Drizzle ORM Migrations (Primary)
Generated by `drizzle-kit` from the Drizzle schema files in `lib/db/src/schema/`. Applied automatically when the API server starts up (via `migrate()` in the server's `index.ts`).

### 2. Supplemental Migration Scripts (Manual)
For schema additions made after initial Drizzle migration setup (e.g., new columns added to existing tables), a manual migration script exists at [`lib/db/migrate-store-db.cjs`](file:///d:/Erp/ERP/lib/db/migrate-store-db.cjs).

Run it with:
```bash
node lib/db/migrate-store-db.cjs
```

**What `migrate-store-db.cjs` migrates (idempotent ÔÇö safe to re-run):**
1. Creates `treasury_transfers` table if missing
2. Creates `treasury_adjustments` table if missing
3. Adds `salary_records.pay_period_type` column if missing (DEFAULT `'MONTHLY'`)
4. Adds `salary_records.advance_deduction` column if missing (DEFAULT `'0'`)
5. Adds `salary_records.other_deductions` column if missing (DEFAULT `'0'`)

> As of the last migration run (2026-07-14), all of the above are confirmed present in the production database.
